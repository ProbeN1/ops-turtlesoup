# Scenario Authoring

本文档用于编写和审查运维故障海龟汤题目，尤其是玩家首先看到的“汤面”。

## 汤面生成提示

目标：让玩家像刚接手一个陌生故障的值班运维一样，只看到有限、精确、带一点混乱感的表象。

生成汤面时遵循：

- 控制在 1-2 句短句。
- 只写第一眼能看到的症状、告警、仪表盘异常或交接信息。
- 必须包含可观测的精确细节：时间点、指标名、变化前后数值、持续时间；如果合适，再加一个小的交接事实。
- 不要直接点出真实子系统，除非它在接手时已经明确可见。
- 不要提前给完整时间线、最近变更、日志细节、多个指标或已执行处置，除非少量线索缺失会导致题目不可解。
- 保持“刚接班一脸懵”的现场感，例如：刚被电话叫醒、群里只丢了一张截图、监控全红但没人知道为什么、上一班说重启没用。
- 精确信息只能描述观察结果，不能把根因编码进汤面。

示例：

```text
00:03 被电话叫醒，`billing-vm-02:/data` 使用率从 78.4% 升到 96.8%，00:14 告警自动恢复到 79.1%，期间业务错误率一直低于 0.1%。
```

## 难度风格

- easy：一个主要故障机制、常见运维域、线索直接，适合新手。
- medium：两个因素互相作用，至少一个误导表象，需要玩家缩小范围。
- hard：多层因果链，包含时间、依赖或状态陷阱，需要区分症状和根因。

## 成功标准

- easy：玩家说出核心根因，并覆盖至少一个关键定位点。
- medium：玩家说出根因，并至少说明两个关键关联点，例如范围、触发条件、误导表象、处置为什么表现异常。
- hard：玩家说明端到端因果链，包括触发事件、放大机制、关键依赖或状态问题，以及为什么恢复或回滚存在滞后。

不要因为玩家只猜到组件名、方向或表面症状就判定成功。玩家在多轮对话中已经累计拼齐关键事实时，即使最后一句只补上最后缺口，也可以判定成功。

## 标准题目模板

```json
{
  "id": "easy-001",
  "title": "",
  "difficulty": "easy | medium | hard",
  "category": "",
  "tags": [],
  "infra_background": {},
  "story": "",
  "answer": "",
  "must_discover": [],
  "misleading": [],
  "forbidden": [],
  "question_rules": {
    "yes": [],
    "no": [],
    "irrelevant": []
  },
  "thinking_path": [],
  "root_cause": "",
  "temporary_fix": "",
  "permanent_fix": "",
  "knowledge_points": [],
  "references": []
}
```

## 基础设施背景

`infra_background` 应描述玩家揭晓后理解故障所需的系统背景，但不应直接泄露根因。推荐写成对象，字段可包括：

- `platform`
- `architecture`
- `database`
- `storage`
- `observability`
- `traffic`
- `scheduler`
- `dependency`

揭晓接口会同时返回可直接展示的 `infraBackground` / `infraBackgroundText` 文本，以及保留结构的 `infraBackgroundRaw` / `infra_background` 对象。前端应优先展示文本字段，只有文本缺失或异常时才格式化原始对象。
